# 需求规约模板：大模型 API 高并发调用、限流熔断与数据清洗规约 (LLM API High-Concurrency Sync)

## 1. 需求背景 (Background)
发包方计划对海量的业务文本（如客服对话历史、商品评论、非结构化文档）使用大语言模型（如 GPT-4、Claude、DeepSeek）进行批量分类、摘要提取或格式化抽取。由于待处理数据量达数万条，单线程同步调用极慢，需采用并发请求。然而，高并发调用极易触发大模型 API 的速率限制（Rate Limits: TPM / RPM），需设计一套可靠的带有令牌桶限流、错误退避、断点续传及格式清洗的分布式并发调用中间件。

---

## 2. 非目标 / 不承接范围 (Out of Scope)
*   **模型精调与基础架构优化**：不提供开源大模型的本地 LoRA 精调、推理加速（如 TensorRT-LLM）或私有化部署调优。
*   **接口账号越权与黑产调用**：不承接任何使用非法逆向 API、盗刷账号、利用接口漏洞逃避平台正常计费的业务逻辑。
*   **长文本生成洗稿**：不承接旨在批量生成低质量垃圾内容、规避搜索引擎原创检测的“洗稿”黑产流程。

---

## 3. 输入与输出规范 (Inputs & Outputs)
*   **输入端数据源**：
    *   本地数据库（如 PostgreSQL/SQLite）中存放的待处理文本数据集，包含唯一主键 `task_id` 和待清洗文本 `raw_content`。
    *   API 秘钥配置：发包方提供的合法大模型 API Key（通过环境变量注入，不允许硬编码写入源码）。
*   **输出端目标**：
    *   将大模型返回的 JSON 结构化数据清洗解析，写入目标数据表。
    *   清洗控制：如果模型返回的数据包含 Markdown 代码块标记（如 ```json ... ```），程序必须实现自动剥离与 JSON 解析防错。

---

## 4. 验收标准 (Acceptance Criteria)
*   **并发限流可靠性**：配置并发线程数为 10 时，系统能够自动监控 API 返回 of HTTP 429 (Too Many Requests) 响应，并自动读取 `Retry-After` 头部进行指数退避延迟重试，不产生请求死锁。
*   **断点续传能力**：在执行中途强行终止进程后，再次启动程序时，系统应自动跳过数据库中状态已标记为 `SUCCESS` 的数据行，仅对未处理或失败的记录进行增量续传。
*   **解析容错率**：对于模型偶发生成的非标准 JSON（如末尾缺少逗号、含有控制字符），系统需具备基础的正则清洗和错误回退处理，确保单次调用失败率控制在 1% 以下。

---

## 5. 风险边界与费用约定 (Risk Boundaries & Cost Agreements)
*   **运行 Token 费率上限**：发包方需自知大模型调用按 Token 计费。服务商不承担由于高频并发调用产生的任何 API 账单费用。发包方可在配置文件中设定每日/单次任务的“消费熔断上限”（以美元/人民币计），一旦超出额度立即停止任务。
*   **模型幻觉与输出偏离**：由于大模型本身存在“幻觉”（Hallucination），服务商无法保证大模型返回结果的 100% 业务准确度，发包方需自行提供验收判定 Prompt 并为最终输出语义负责。

---

## 6. 可交付物列表 (Deliverables)
*   `[ ]` 基于 Python (Asyncio) 或 Node.js 的高并发 API 批量调用脚本源码
*   `[ ]` 带有令牌桶（Token Bucket）的请求限流器组件
*   `[ ]` 数据库断点续传与任务日志存储表设计 (DML)
*   `[ ]` 配置文件模板（包含并发数、最大重试次数、API 熔断费率上限参数）

---

## 7. 适合 AI Agent 执行的任务切片 (Agent Task Slices)
1.  **异步限流包装器编写**：让 Agent 自动编写一个 Python `semaphore` 异步并发控制装饰器，将任何普通 API 请求函数包装为支持指数退避的重试流。
2.  **Markdown-JSON 剥离器**：使用 Agent 编写高鲁棒性的文本解析函数，能够兼容提取并解析大模型返回的带有各类前导/后缀字符的 JSON 字符串。
3.  **单元测试 Mock 服务器**：让 Agent 用 FastAPI 搭建一个本地 API Mock 服务器，模拟大模型 API 随机返回 HTTP 429/500/200 响应，以测试高并发组件的稳定性。
