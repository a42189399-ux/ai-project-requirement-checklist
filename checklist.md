# AI/自动化项目发包前需求规约 Checklist (完整指南)

本指南针对中小型 AI 项目、数据自动同步工作流或小工具在委托第三方开发前，如何编写规范的技术说明书（PRD）提供工程化的对齐指标。

---

## 🔍 需求规约 10 个核心核对项

### 1. 核心输入边界 (Input Specification)
发包前必须明确数据的源头。
*   **格式核对**：数据是零散的文本、结构化的 JSON 字符串、PDF 附件、阿里云盘的分享链接，还是通过 API 实时下发的流？
*   **频次核对**：预计系统运行的调用频率是怎样的？是每秒 10 次的并发，还是每小时定时跑一次的批量任务？
*   **体积核对**：单个输入文件的大小是多少？（如：PDF 文件平均在 10MB 以内，大模型 Token 限制在多少字内？）

### 2. 期望输出边界 (Output Specification)
定义项目最终达成动作的“物理状态”。
*   **动作核对**：系统如何通知或展现成果？例如：“在面包多平台监听到付款后，将一条激活码自动填入买家浏览器端，并同步在 MySQL 数据库记录已售出。”
*   **存储核对**：处理完毕的数据需要转存到哪里？（如：自建的 PostgreSQL 数据库、特定的飞书多维表格、还是通过 webhook 回传给第三方业务系统？）

### 3. 网关与 API 拥有权 (API & Gateways)
明确费用主体与限流。
*   **拥有权核对**：开发中所用到的付费 API（如 OpenAI API Key、Claude API Key、阿里云短信网关、Feishu 开放平台自建应用）账号必须由发包方自行申请并拥有，开发方仅使用由你下发的测试 Key。
*   **限流核对**：是否已告知开发方你所拥有账号的 API QPS/RPM 频次限制（如每分钟限制请求 3 次），并要求其在代码中编写队列或延迟重试机制？

### 4. 运行环境与托管成本 (Environment & Hosting)
规避后期隐藏的运维费用。
*   **托管位置**：系统最终运行在何处？是托管在境外 Vercel/Supabase（免备案但需要网络优化），还是国内腾讯云宝塔面板（需要网站 ICP 备案）？
*   **成本核对**：系统需要的服务器、RDS 数据库、Redis 缓存的月度/年度账单预算是否已经在你的商业计划中？

### 5. 明确的验收判定标准 (Acceptance Criteria)
项目交付的核心红线。
*   **成功率指标**：什么是交付的“标准通过线”？（如：“利用 AI 进行简历解析时，20 份测试简历的字段提取成功率必须 $\ge 95\%$，且没有严重格式错乱，判定为一期交付合格。”）
*   **红线测试集**：发包方应当在本地预留 5-10 条包含各种奇葩格式的真实数据，不提前发给开发商作为开发数据，仅作为最后一轮验收时的红线测试。开发商必须通过此红线测试才可结款。

### 6. 异常边界与降级机制 (Error Handling & Failover)
AI 系统常因上游接口超时或幻觉出错，系统必须有容错。
*   **重试与限流**：上游 API 报错时，系统应当自动重试几次？重试的时间间隔是怎样的？
*   **降级策略**：若持续失败，系统是静默报错，还是自动在飞书/钉钉群触发 Webhook 发送一条“XX订单自动发货失败，请管理员人工核销”的警报？

### 7. 数据存储与安全红线 (Data Security)
*   **敏感信息加密**：数据库中存储的用户手机号、邮箱、特别是第三方 API Key 是否必须进行单向哈希或对称加密存储？
*   **密钥管理**：严禁开发人员在代码中硬编码任何 API Key，所有密钥必须从服务器环境变量（ENV）或安全 Key 仓库中读取。

### 8. Token 费率与持续运行成本 (Token Rate & Operating Cost)
*   **单次成本估算**：调用一次 AI 接口（如：单条简历分析），在输入和输出平均 Token 数下，消耗的直接代币成本是多少？
*   **定价覆盖率**：如果单次运行消耗 $0.1 的 API 费用，而你计划每次向用户收取 0.5 元人民币，该商业闭环将直接亏损。

### 9. 界面与人机交互边界 (UI & Interaction)
*   **展现形式**：这是一个纯后台的数据自动流转工作流（通过 cron 定时或 webhook 触发），还是需要一套可供用户登录的 Web/微信小程序界面？
*   **原型对齐**：如果是 UI 页面，建议使用现成的前端开源框架默认样式，不搞复杂的高度定制化设计，以节约首期开发费用。

### 10. 交付物归属与工程规范 (Code Delivery & DevOps)
*   **源码托管**：源码必须从第一天起就提交并在发包方自行拥有的私有 GitHub 仓库中，由发包方掌握项目核心资产主权。
*   **自动化部署**：开发方是否需要配置 GitHub Actions 或 Webhook，实现代码向生产服务器的自动化一键打包部署（CI/CD）？
*   **技术文档**：是否要求开发方提供一期《系统时序与数据流架构图.png》与《部署与运维指南.md》？
